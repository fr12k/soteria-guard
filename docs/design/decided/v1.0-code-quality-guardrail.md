## 1. Motivation & Core Principle

### Problem

Most static analysis tools fail in practice because they measure **all
code equally**, flooding teams with noise. A 5000-line file that nobody
ever touches is structurally ugly but behaviorally irrelevant. A
200-line file that every sprint collects new edge-cases is a ticking
time bomb.

### Principle

> Complexity only matters if you need to deal with it.

The guardrail fuses **static structure** with **evolutionary behavior
from git history**. A file that is *complex* AND *changed often* AND
*touched by many authors* AND has a *rising complexity trend* -- that
file is a **hotspot**, and that is where the bodies are buried.

### Book Reference

Adam Tornhill, *Your Code as a Crime Scene* (Pragmatic Bookshelf, 2015)
-- the metaphor is forensic psychology: VCS logs are the crime scene
record, and we profile the *offender* (problematic code) through
geographical offender profiling techniques.

## 2. Metric Taxonomy

### 2.1 Static Structure (Current Snapshot)

All metrics are language-independent -- they operate on raw text and
file-tree layout.

| Metric | What It Measures | Implementation |
|---|---|---|
| **Lines of Code (LOC)** | File size, excluding blanks and pure-comment lines | Line counting with skip rules |
| **Indentation Complexity** | Mean nesting depth across all lines in a file | Count leading whitespace; see §6 |
| **Max Nesting Depth** | Deepest single conditional/loop nest in the file | Max of per-line indent values |
| **Indentation StdDev** | How uneven the nesting is across the file | Standard deviation of indent depths |
| **Comment Ratio** | Comment lines / total lines (proxy for self-documentation) | Regex for comment markers |
| **File Count per Module** | How many source files a directory / component contains | File-tree walk |

### 2.2 Evolutionary Behavior (Git History)

| Metric | What It Measures | Source Command |
|---|---|---|
| **Revision Count** | How many commits touched this file | `git log --oneline -- <file> \| wc -l` |
| **Author Count** | How many distinct developers touched it | `git shortlog -sne -- <file>` |
| **Total Churn** | Lines added + lines deleted (absolute) | `git log --numstat -- <file>` |
| **Entity Effort** | Fraction of total project churn in this file | Sum(churn per file) / total churn |
| **Main Developer %** | Share of commits by the single most active author | max(author_commits) / total_commits |
| **Change Coupling** | Pairs of files that co-change in the same commits | `git log --name-only` → co-occurrence matrix |
| **Complexity Trend** | Slope of indentation complexity over a sliding window | Complexity per revision → linear regression |

### 2.3 Why No Cyclomatic Complexity?

Cyclomatic complexity requires a language-specific AST parser. The
indentation proxy (§6) achieves ~90% correlation with McCabe's metric
(ICPC 2008) without any language knowledge. This keeps the tool truly
polyglot.

## 3. Derived Signals

The power is not in any single metric but in their combinations.

### 3.1 Hotspot Score

```
hotspot_score = revision_count × mean_indentation_complexity
```

**Interpretation**: Files with high scores are expensive -- they are
structurally complex AND require frequent attention. These are the
highest-ROI refactoring targets.

**Origin**: Tornhill's "offender profile" -- the overlap of complexity
and change frequency. Typically 4–6 % of files account for most
maintenance cost.

### 3.2 Knowledge Loss Risk

```
knowledge_loss_risk = mean_indentation_complexity × (1 - main_dev_ratio)
```

**Interpretation**: High complexity with low single-developer ownership
means the code is hard to understand AND nobody has a complete mental
model. If the main contributor leaves, the rest of the team inherits
an incomprehensible module.

### 3.3 Developer Congestion

```
developer_congestion = author_count / revision_count
```

**Interpretation**: Values approaching 1.0 mean "every revision has a
different author" -- a coordination bottleneck. Empirical research
shows a strong correlation between high congestion and defect density.

### 3.4 Coupling Strength

```
coupling_degree_ab = shared_commits_ab / total_commits_a
```

**Interpretation**: If file A and file B co-change in >30 % of A's
commits, they are logically (temporally) coupled. This is an
architectural signal -- the coupling may be intentional (legitimate
dependency) or incidental (copy-paste, missing abstraction).

### 3.5 Complexity Trend

```
complexity_trend = slope(linear_regression(
    x = revision_timestamps,
    y = mean_indentation_complexity
))
```

**Interpretation**:
- **Flat or falling** → code is stable or improving (green).
- **Rising** → code accumulates complexity faster than new lines
  (deteriorating -- red). This is the earliest warning signal.

## 4. Threshold Zones

Every metric uses **percentile-based thresholds calibrated to the
project's own data**, not arbitrary magic numbers.

### 4.1 Zone Definitions

| Zone | Label | Percentile Range | Meaning | Recommendation |
|---|---|---|---|---|
| **1** | 🟢 **Green** | 0–60th pct | Healthy, low risk | No action needed |
| **2** | 🟡 **Yellow** | 60–85th pct | Moderate; watch | Review if trend is rising |
| **3** | 🟠 **Orange** | 85–95th pct | Problematic | Plan refactoring |
| **4** | 🔴 **Red** | 95–100th pct | Critical | Refactor urgently |

### 4.2 Why Percentile-Based?

- **Auto-adapts** to project size, language, and team norms.
- **No false positives** from arbitrary thresholds (e.g., "max 20 methods
  per class" that flag a generated file).
- **Comparable across projects** — zone 4 always means "top 5% of this
  project", regardless of absolute numbers.

### 4.3 Per-Metric vs Composite

Each file gets a per-metric zone. The composite (overall) zone is the
**worst** per-metric zone. The report lists each metric separately
so the developer knows *why* the file was flagged.

### 5.2 Scanning Strategies

| Mode | When | Scope | Cost |
|---|---|---|---|
| **Full scan** | On demand or scheduled | Entire repo (last N months) | Slow but exhaustive |
| **Incremental scan** | On demand | Append new commits since last scan | Fast |
| **Ad-hoc** | On demand | Any file or commit range | Flexible |

The tool always recomputes project-wide percentiles from the scanned
data. There is no separate "PR mode" — the binary analyzes what it is
told to analyze and writes results to stdout/files.

### 5.3 Single-Pass Git Log Parsing

A single `git log` invocation streams all the data needed for both
static and evolutionary analysis:

```
git log --after=<date> --numstat --pretty="commit %H %ai %an"
```

The output is parsed line-by-line. No file is read from disk until
the static-metrics phase, which is file-system driven.

## 6. Language-Independent Complexity Proxy

### 6.1 The Indentation Method

Per Tornhill / *Reading Beside the Lines: Indentation as a Proxy for
Complexity Metric* (ICPC 2008), we use leading whitespace as a proxy
for cyclomatic complexity.

**Algorithm**:

```
For each line in the file:
    strip = line.lstrip()
    if strip is empty:               skip
    if strip starts with comment
      marker (#, //, --, %, ;, '):  skip
    indent = len(line) - len(strip)
    normalize: indent // 4           (treat tab = 4 spaces)
    append to depths[]

Return:
    mean = avg(depths)
    max  = max(depths)
    std  = stdev(depths)
    total = sum(depths)
```

### 6.2 Research Validation

The ICPC 2008 paper demonstrated that indentation depth correlates with
cyclomatic complexity at r ≈ 0.90 across Java, C, and C++ codebases.
It is the **only language-independent proxy with published empirical
validation**.

### 6.3 Limitations

- Does not capture non-linear control flow (exceptions, gotos, etc.).
- Skewed by generated code (protobuf, parser generators) -- those files
  should be excluded via `.guardrailignore`.
- Tabs vs. spaces ambiguity: we normalize on assumption that one tab
  equals 4 spaces; configurable.

## 7. Self-Calibrating Thresholds

### 7.1 Data Model

The history DB is an append-only JSON-lines file:

```jsonl
{"scan_id": "2025-06-16T06:00Z", "file": "src/parser.zig",
 "date": "2025-06-16", "loc": 340, "indent_mean": 2.8, "indent_max": 9,
 "revisions": 14, "authors": 3, "main_dev_pct": 0.62, "churn": 420}
```

### 7.2 Percentile Calculation

On each scan, compute percentiles across the current snapshot:

```
sorted_scores = sorted([f.hotspot_score for f in all_files])
p60 = sorted_scores[floor(0.60 * n)]
p85 = sorted_scores[floor(0.85 * n)]
p95 = sorted_scores[floor(0.95 * n)]
```

These percentiles are **snapshotted to JSON** and used for all zone
assignments in that report.

### 7.3 Decay

Old data in the history DB (older than 12 months) is archived or
dropped. The active window is:

- **Git history scan**: last 6 months (configurable)
- **Trend analysis**: last 12 months (but 6-month window for recency)

This prevents ancient hotspots (since fixed) from skewing current
thresholds.

### 7.4 Initial Bootstrap

On first run (no history DB exists):
1. Run a full scan immediately
2. Use conservative fallback thresholds:
   - Hotspot score > 50 → red
   - Indent max > 8 → red
   - Author count > 5 → orange
3. After the first scan, switch to percentile-based

## 8. Change Coupling (Temporal Coupling)

### 8.1 Definition

Two files are *change-coupled* (or *temporally coupled*) if they are
modified together in the same commits more often than expected by
chance. This is the closest we can get to detecting logical dependencies
without a language-specific dependency graph.

### 8.2 Algorithm

```
For each commit in the history window:
    files = set of files modified in that commit
    for each pair (a, b) in files × files, a ≠ b:
        increment shared_commits[a][b]
        increment total_commits[a]
        increment total_commits[b]

For each pair (a, b):
    degree_ab = shared_commits[a][b] / total_commits[a]
    degree_ba = shared_commits[a][b] / total_commits[b]
    coupling_strength = max(degree_ab, degree_ba)
```

For performance, the matrix is stored sparsely: only pairs with
`shared_commits ≥ 3` and `coupling_strength ≥ 0.15` are kept.

### 8.3 What It Detects

| Pattern | Coupling Signature | Likely Cause |
|---|---|---|
| Copy-paste clone | High degree (≥0.5), no import | Duplicated logic in unrelated modules |
| Missing abstraction | Medium-high, files in different packages | Should be unified behind an interface |
| Inadequate encapsulation | High degree, one direction | Changing A always needs changes to B |
| Legitimate co-evolution | Medium, both sides, clear API boundary | Interface + implementation, expected |
| Temporal accident | Low, sporadic | Noise; ignore |

### 8.4 Trend Annotation

Each coupling pair is annotated with a trend: is the coupling getting
**stronger** (red), **weaker** (blue), or **stable** (yellow)? This is
computed by splitting the history into two equal windows and comparing
the coupling degree in each.

## 9. CLI Interface

### 9.1 Usage

```
soteria [options] [path]

Analyze a codebase and produce a quality report.

Arguments:
  path                  Path to the git repository root (default: ".")

Options:
  --after <date>        Git history window (e.g., "6 months ago", "2024-01-01")
                        (default: "6 months ago")
  --out <file>          Write machine-readable report (JSON) to file
  --out-markdown <file> Write human-readable report (Markdown) to file
  --history <file>      Path to history DB (JSON lines) for trend tracking
                        (default: .soteria/history.jsonl)
  --coupling <file>     Path to write coupling matrix (JSON, sparse)
                        (default: .soteria/coupling.json)
  --thresholds <file>   Path to write threshold snapshot (JSON)
                        (default: .soteria/thresholds.json)
  --ignore <file>       Path to .guardrailignore (default: .guardrailignore)
  --verbose             Print progress to stderr
  --version             Print version and exit
  --help                Print help and exit
```

### 9.2 Exit Codes

| Code | Meaning |
|---|---|
| 0 | Success; no critical issues |
| 1 | Analysis completed, but one or more files in zone 4 (red) |
| 2 | Analysis error (bad path, no git repo, parse failure) |

### 9.3 Outputs

The tool always prints a summary to stdout. It writes four optional
output files:

1. **JSON report** (`--out`) — machine-readable, structured data
2. **Markdown report** (`--out-markdown`) — human-readable, formatted
3. **History DB** (`--history`) — append-only JSON lines, updated
4. **Coupling matrix** (`--coupling`) — sparse JSON, precomputed
5. **Threshold snapshot** (`--thresholds`) — JSON, current percentiles

### 9.4 `.guardrailignore` Format

```
# Lines starting with '#' are comments
# Glob patterns, one per line

vendor/**
generated/**
*.pb.go
*.miranda.zig
dist/**
node_modules/**
```

Files matching any pattern are skipped entirely (not scored, not
included in percentile calculation).

## 10. Implementation Components

### 10.1 Module Map

All components are Zig source files within a single binary. No separate
processes, no daemon, no server.

| Module | Responsibility | Est. LOC |
|---|---|---|
| `main.zig` | CLI argument parsing, orchestration of pipeline | 120 |
| `git_log.zig` | Invoke `git log`, stream-parse output into per-file time series | 150 |
| `complexity.zig` | Indentation analysis per file (mean, max, std, total) | 100 |
| `churn.zig` | Revision count, author count, main-dev %, churn aggregation | 100 |
| `coupling.zig` | Co-change matrix from commit data (sparse HashMap) | 180 |
| `trend.zig` | Linear regression over complexity time series | 80 |
| `thresholds.zig` | Percentile calculation (sort + interpolation) + zone assignment | 90 |
| `signals.zig` | Derived signal computation (hotspot, congestion, risk, etc.) | 80 |
| `report.zig` | Markdown report generation, JSON report generation | 120 |
| `history.zig` | JSON-lines history DB read/append | 80 |
| `ignore.zig` | `.guardrailignore` parsing + glob matching (`*`, `**`, `?`) | 100 |
| `scan.zig` | File-tree walk, source-extension filter, scan orchestration | 80 |

**Total**: ~1280 LOC core tooling.

### 10.2 Dependencies

| Dependency | Use | Rationale |
|---|---|---|
| Zig 0.13+ | Compiler and standard library | Core language; ships with JSON, HTTP, sort, HashMap |
| Git CLI | Data source | Already present on any development/CI machine |

**Zero external library dependencies.** The standard library provides:
- `std.json` — JSON parsing and writing for reports, history DB,
  coupling matrix, and thresholds snapshot
- `std.sort` — percentile calculation via intro-sort
- `std.hash_map` — sparse coupling matrix and per-file state
- `std.process.Child` — git CLI invocation
- `std.fs` — file-tree walk for source discovery
- `std.unicode` — UTF-8 aware line handling

No AST parsers, no language-specific SDKs, no databases, no HTTP
libraries, no math libraries.

## 12. Pragmatic Decisions

### 12.1 What We Measure vs. What We Skip

| Skipped Metric | Why Excluded |
|---|---|
| Cyclomatic complexity | Requires AST per language; indentation proxy works without it |
| LCOM4 (cohesion) | Requires class/method parsing per language |
| Fan-in / fan-out | Requires dependency graph per language |
| Cognitive complexity | Subjective; SonarQube-specific; per-language rules |
| Test coverage | Dynamic metric, not static; different tool |
| Code duplication (token-level) | Already captured by temporal coupling (files that co-change suspiciously = likely clones) |
| Commit message sentiment | High overhead for marginal signal |
| Halstead metrics | Weak empirical correlation with defects |

### 12.2 Cost/Benefit Tradeoffs

- **No language parser**: saves 90 % of implementation effort and makes
  the tool instantly usable on any project. Cost: cannot detect
  function-level smells (long parameter lists, god functions by name).
  Mitigation: file-level indentation max and LOC catch most god
  functions.
- **Git-only VCS**: Code Maat supports hg, svn, p4, tfs, but git
  covers >90 % of modern projects. Extending later is straightforward.
- **6-month window**: long enough to show trends, short enough to
  ignore fixed hotspots. Tune via config.
- **Percentile thresholds over absolute**: avoids "one-size-fits-all"
  nonsense. A 1000-line embedded C file is normal; a 1000-line Ruby
  file is a red flag. Percentiles handle this automatically.
- **Single binary**: no runtime, no dependencies, no install steps.
  Download and run.
- **CLI-first**: the tool reads a directory and writes files. It does
  not talk to any API, post to any service, or require any platform.
  Integration is left to the caller (GitHub Actions, Makefile, cron).

## 14. Limitations & Known Gaps

### 14.1 False Positives

- **Generated code** that isn't in `.guardrailignore` will score high
  on complexity. Fix: keep the ignore list up to date.
- **Configuration files** with deep YAML/JSON nesting will score as
  "complex" though they are data, not logic. Fix: only scan source code
  extensions (`.zig`, `.rs`, `.go`, `.java`, `.js`, `.ts`,
  `.c`, `.h`, `.cpp`, `.py`, `.rb`).
- **Monolithic commits** (rename, reformat) skew churn and coupling.
  Fix: detect commits that touch >50 files and exclude them or flag
  them as "non-functional".

### 14.2 False Negatives

- **Pure cyclomatic complexity** that doesn't show up in indentation
  (e.g., early-return patterns with flat nesting but many branches).
  Mitigation: accept the proxy limitation; add language-specific
  plugins later if needed.
- **Runtime coupling** (files that must be deployed together but are
  never changed in the same commit). Mitigation: cannot detect from VCS
  alone; requires deployment analysis (out of scope).
- **Small-but-critical files** with low LOC and few revisions but
  catastrophic impact if wrong. Mitigation: add a manual override for
  designated critical files.

### 14.3 Operational Risks

- **History DB grows unbounded**. Mitigation: archive entries older
  than 12 months; keep only aggregations for older data.
- **Git log parsing is slow on large repos** (>50k commits). Mitigation:
  incremental parsing (store last parsed commit hash, only parse new
  commits).
- **CI timeout on large repos**. Mitigation: use `--after` to narrow
  the window; run full scans on a schedule, not per-PR.

## 15. Glossary

| Term | Definition |
|---|---|
| **Hotspot** | A file with high hotspot score (revisions × complexity). Primary refactoring target. |
| **Change coupling** | Two files that frequently change together. Syn: temporal coupling, logical coupling. |
| **Degree** | Coupling strength: shared commits / total commits. Range 0–1. |
| **Churn** | Lines added + lines deleted. Measures how much code movement a file undergoes. |
| **Entity effort** | Fraction of total project churn attributable to one file. |
| **Main developer %** | Proportion of commits by the single most active author. |
| **Developer congestion** | Number of distinct authors per revision. High values indicate coordination bottlenecks. |
| **Complexity trend** | Slope of indentation complexity over time. Rising = deteriorating. |
| **Knowledge loss risk** | Complexity × (1 − main developer %). High = bus factor risk. |
| **Zone** | Classification bucket (1–4) based on project-specific percentiles. |
| **Self-calibrating** | Thresholds derived from the project's own data, not external standards. |

## 16. Zig-Specific Design Decisions

### 16.1 Language Choice

Zig was chosen for the following reasons:

| Reason | Detail |
|---|---|
| **Single static binary** | No runtime, no interpreter, no VM. A single `x86_64-linux` binary can be dropped into any CI pipeline or developer machine. |
| **Zero dependencies** | The standard library provides JSON, sorting, hashing, processes, file I/O, and math. No `numpy`, no `pyyaml`, no `requests`. |
| **Explicit memory management** | Arena allocators for analysis-run-scoped data; no GC pauses; predictable memory footprint (~1–5 MB for a medium repo). |
| **Cross-compilation** | Build for `x86_64-linux`, `aarch64-linux`, `x86_64-macos`, `aarch64-macos` from a single machine. |
| **Error handling** | Zig's error union type forces proper handling of I/O, parsing, and subprocess failures — no silent fallbacks. |

### 16.2 Memory Model

The tool uses a two-level allocation strategy:

1. **Arena allocator** (per analysis run): all per-file metrics,
   coupling pairs, and report state are allocated into a single arena.
   When the report is written, the arena is freed in one shot. No
   individual frees needed.

2. **General-purpose allocator** (for long-lived state): the history DB
   cache, threshold snapshot, and CLI state outlive a single scan.

This avoids fragmentation and keeps allocation overhead negligible.

### 16.3 Subprocess Strategy

Git is invoked via `std.process.Child` with piped stdout. The output is
read line-by-line as it streams — no temporary files, no buffering the
full output. This keeps memory constant even for repos with 50k+
commits.

### 16.4 Coupling Matrix Storage

The coupling matrix is a `std.StringHashMap` keyed by a packed `u64`
pair (file index A, file index B). File paths are interned into a
deduplicated string table during git log parsing. A sparse threshold
(`shared_commits < 3` or `degree < 0.15`) filters entries before they
enter the map, keeping the structure small.

### 16.5 JSON Serialization

All JSON output (report, history DB, coupling matrix, thresholds) uses
`std.json.WriteStream` with a `std.ArrayList(u8)` buffer. History DB
entries are written one line at a time with `\n` separators (JSON
lines format). Reading is done with `std.json.Scanner` in streaming
mode — no full-file parse.

### 16.6 Glob Matching

The `.guardrailignore` parser implements a minimal glob dialect:
- `*` matches any sequence of non-separator characters
- `**` matches any sequence including separators (directory wildcard)
- `?` matches a single non-separator character
- `#` starts a comment line

This is ~100 LOC, avoids pulling in a regex engine, and covers all
practical ignore patterns.
