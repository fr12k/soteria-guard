# HotSpot Score Storage File

**Status:** implemented
**Affects:** `src/types.zig`, `src/main.zig`, `src/report.zig`
**Related:** `v1.0-code-quality-guardrail.md` (§3.1, §13)

## Problem Statement

Hotspot scores are currently embedded inline in the full JSON report
(`--out` flag) but have no dedicated persistent storage. Consumers
that only need hotspot data (CI dashboards, IDE plugins, PR comments)
must parse the entire report, which is expensive and couples them to
the full schema.

Meanwhile, the history file (`history.jsonl`) stores raw components
(`revisions`, `indent_mean`) but not the derived score. To answer
*"is this file's hotspot score trending up?"* a consumer must
re-compute the score from raw fields — and the `HistoryEntry` schema
excludes other derived signals like `knowledge_loss_risk` and
`developer_congestion`, so those trends are invisible historically.

## Design Decisions

### Decision 1: Dedicated `hotspot.json` file

**Yes, add a dedicated `.soteria/hotspot.json`.** One object per scan
that maps file paths to their current hotspot scores + zones, plus a
top-level metadata block. This gives consumers a ~2 KB file instead
of a potentially multi-MB full report.

**Why not just use the existing report?**
- The report bundles couplings, all six per-file zones, and full
  evolutionary metrics. A CI job that only wants to fail on red
  hotspots should not parse that.
- The report is a one-shot artifact (written to `--out` if the flag
  is given). The hotspot file lives in `.soteria/` alongside
  `thresholds.json` and `coupling.json` — always written, always
  available.

### Decision 2: Extend HistoryEntry with derived signals

**Yes, add `hotspot_score`, `knowledge_loss_risk`, and
`developer_congestion` to `HistoryEntry`.** Currently the history
file stores raw components (`revisions`, `indent_mean`, `authors`,
etc.) and the consumer must re-derive the signals. Pre-computing them
at write time costs nothing (the signals are already computed during
the scan) and makes historical queries O(1) instead of O(fields).

Raw fields are **retained** alongside the derived signals. They
enable independent analysis (e.g., "revisions spiked but complexity
stayed flat") and cost almost nothing to store.

### Decision 3: Two-file strategy — snapshot JSON + history JSONL

`hotspot.json` is a **snapshot** — overwritten each scan. Compact,
predictable, single-parse for CI/IDE.

The existing `history.jsonl` (extended in Decision 2 with
`hotspot_score`) serves as the **historical record**. No separate
`hotspot.jsonl` is needed: every file's hotspot score over time is
already recorded in `history.jsonl` once the new fields are added.

This gives us both access patterns without a third file.

### Decision 4: No new CLI flag

The file always goes to `.soteria/hotspot.json`. If a consumer wants
a custom path, they can use the full `--out` report. Adding a
`--hotspot` flag creates unnecessary surface area for a file that has
one use case.

### Decision 5: `scan_id` — use ISO timestamp

Currently `scan_id` is hardcoded as `"live"`. The hotspot file
(and by extension the report) should use an ISO-8601 timestamp
(e.g., `2025-05-19T12:00:00Z`) as the scan identifier. This makes
it possible to correlate hotspot snapshots across time without
parsing the file contents.

**Implementation**: generate once at the start of `main()` and
thread through to all outputs. A simple approach:

```zig
var now_buf: [24]u8 = undefined;
const scan_id = std.fmt.bufPrint(&now_buf, "{d:0>4}-{d:0>2}-{d:0>2}T{d:0>2}:{d:0>2}:{d:0>2}Z", .{
    now.year, now.month, now.day,
    now.hour, now.minute, now.second,
}) catch "live";
```

This is a one-line change in `main.zig` that touches report, history,
hotspot, and thresholds outputs.

## Proposed Schema

### `.soteria/hotspot.json` — Snapshot

```json
{
  "scan_id": "2025-05-19T12:00:00Z",
  "created_at": "2025-05-19T12:00:00Z",
  "thresholds": {
    "p60": 10.0,
    "p85": 30.0,
    "p95": 50.0
  },
  "files": [
    {
      "path": "src/main.zig",
      "hotspot_score": 45.2,
      "zone": "red",
      "zone_level": 4
    },
    {
      "path": "src/utils.zig",
      "hotspot_score": 5.0,
      "zone": "green",
      "zone_level": 1
    }
  ]
}
```

- `files` is sorted descending by `hotspot_score` (highest first).
- Only files with source extensions are included.
- The `thresholds` block is repeated so the file is self-describing
  without needing to read `thresholds.json`.

### Extended `HistoryEntry`

The existing `HistoryEntry` struct gains three new fields:

```zig
pub const HistoryEntry = struct {
    scan_id: []const u8,
    file: []const u8,
    date: []const u8,
    loc: u32,
    indent_mean: f64,
    indent_max: u32,
    revisions: u32,
    authors: u32,
    main_dev_pct: f64,
    churn: u32,
    // NEW — pre-computed derived signals (default to 0.0)
    hotspot_score: f64 = 0.0,
    knowledge_loss_risk: f64 = 0.0,
    developer_congestion: f64 = 0.0,
};
```

The `= 0.0` defaults are critical: existing struct literals in tests
and in `main.zig`'s history write loop compile without changes. Old
`history.jsonl` lines from previous tool versions lack these fields;
when read back, they parse as `0.0` (graceful downgrade — scores of
0 mean "no history" which is safe as a fallback).

## Implementation Plan

### Step 1: Update `HistoryEntry` type (types.zig)

Add three new fields with `= 0.0` defaults:

```zig
pub const HistoryEntry = struct {
    scan_id: []const u8,
    file: []const u8,
    date: []const u8,
    loc: u32,
    indent_mean: f64,
    indent_max: u32,
    revisions: u32,
    authors: u32,
    main_dev_pct: f64,
    churn: u32,
    hotspot_score: f64 = 0.0,
    knowledge_loss_risk: f64 = 0.0,
    developer_congestion: f64 = 0.0,
};
```

### Step 2: Extend `writeHistoryEntry` (report.zig)

Modify the `try w.print("...churn\":{d}}}\n", .{entry.churn});` line
to emit the three new fields after `churn`:

```zig
try w.print(",\"churn\":{d},\"hotspot_score\":{d:.4},\"knowledge_loss_risk\":{d:.4},\"developer_congestion\":{d:.4}}}\n", .{
    entry.churn,
    entry.hotspot_score,
    entry.knowledge_loss_risk,
    entry.developer_congestion,
});
```

### Step 3: Populate derived signals in history write (main.zig)

In the history-write loop (around line 346), the `HistoryEntry`
struct literal needs the three new fields populated from the already-
computed `f.signals`:

```zig
const entry = types.HistoryEntry{
    .scan_id = scan_id,
    .file = f.metrics.path,
    .date = "unknown-date",
    .loc = f.metrics.loc,
    .indent_mean = f.metrics.indent_mean,
    .indent_max = f.metrics.indent_max,
    .revisions = f.evolution.revisions,
    .authors = f.evolution.authors,
    .main_dev_pct = f.evolution.main_dev_pct,
    .churn = f.evolution.churn,
    .hotspot_score = f.signals.hotspot_score,
    .knowledge_loss_risk = f.signals.knowledge_loss_risk,
    .developer_congestion = f.signals.developer_congestion,
};
```

### Step 4: Write `hotspot.json` after zone assignment (main.zig)

After Step 5 (calibrate thresholds, assign zones — around line 215),
before the stdout summary:

```zig
// ── Step 5b: Write hotspot.json ──
{
    const hf = try std.Io.Dir.createFile(repo_dir, io, ".soteria/hotspot.json", .{});
    defer std.Io.File.close(hf, io);

    var buf: [32768]u8 = undefined;
    var w = std.Io.Writer.fixed(&buf);
    const out = &w.interface;

    // Header
    try out.print("{{\"scan_id\":\"{s}\",\"created_at\":\"{s}\"", .{
        scan_id, scan_id,
    });

    // Thresholds block
    try out.print(",\"thresholds\":{{\"p60\":{d:.4},\"p85\":{d:.4},\"p95\":{d:.4}}}", .{
        thresholds.p60_hotspot,
        thresholds.p85_hotspot,
        thresholds.p95_hotspot,
    });

    // Files array — sort descending by hotspot_score
    var sorted_idxs = std.ArrayList(usize).empty;
    defer sorted_idxs.deinit(a);
    for (file_results.items, 0..) |_, i| {
        try sorted_idxs.append(a, i);
    }
    std.mem.sort(usize, sorted_idxs.items, file_results.items, struct {
        fn lessThan(ctx: []types.FileResult, a: usize, b: usize) bool {
            return ctx[a].signals.hotspot_score > ctx[b].signals.hotspot_score;
        }
    }.lessThan);

    try out.writeAll(",\"files\":[");
    for (sorted_idxs.items, 0..) |idx, i| {
        const f = &file_results.items[idx];
        const comma = if (i > 0) "," else "";
        try out.print("{s}{{\"path\":\"{s}\",\"hotspot_score\":{d:.4},\"zone\":\"{s}\",\"zone_level\":{d}}}", .{
            comma,
            jsonEscapeEscaped(f.metrics.path),
            f.signals.hotspot_score,
            f.hotspot_zone.name(),
            @as(u8, @intFromEnum(f.hotspot_zone)),
        });
    }
    try out.writeAll("]}");

    try std.Io.File.writeStreamingAll(hf, io, out.buffered());
}
```

### Step 5: Use real scan_id

In `main.zig`, replace `const scan_id = "live";` with code that
generates an ISO-8601 timestamp. For simplicity, the string is
computed once and shared across all outputs:

```zig
// ISO-8601 timestamp for scan_id — generated once, used everywhere
var scan_id_buf: [24]u8 = undefined;
const scan_id = scan_id_buf[0..0];  // placeholder
```

The exact implementation depends on how `std.time` is accessed in
the current 0.17-dev API. The important invariant: all outputs
(history, report, hotspot, thresholds) share the same `scan_id`
value so they can be correlated.

## Design Decisions Log

### Resolved questions

| Question | Decision | Rationale |
|----------|----------|-----------|
| Include `entity_effort` and `complexity_trend` in hotspot.json? | **No** — hotspot signal only | Consumers that need the full picture should use `--out` report. Keeping hotspot.json focused keeps it small (~2 KB vs multi-MB). |
| Retain raw fields in history after adding derived signals? | **Yes** — retain both | Enables independent analysis ("revisions spiked but complexity stayed flat") at negligible storage cost. |
| Omit hotspot.json when every file is green? | **No** — always write it | CI consumers expect the file to exist. An empty `files` array is valid and signals "nothing to fix". |
| Add a `--hotspot` CLI flag? | **No** — always write to `.soteria/hotspot.json` | One canonical location. Consumers that need a custom path already have `--out`. |
| Decay old history entries when reading? | **Yes** — 12-month window, same as existing | Consistent with `isDateWithin365Days` in `main.zig`. Hotspot.json is a snapshot, so decay applies only to historical trend queries. |
| `scan_id` value? | **Use runtime ISO timestamp** — replace `"live"` | Enables correlating hotspot snapshots across time without parsing content. |

## Backward Compatibility

| Change | Compat? | Notes |
|--------|---------|-------|
| New `hotspot.json` | ✅ | Addition; no existing consumers |
| New fields in `HistoryEntry` | ✅ | Default `= 0.0` in struct definition; old lines parse as `0.0` for missing fields |
| `history.jsonl` format change | ⚠️ | Old lines keep working; new lines have extra fields. Consumers that do strict schema validation (e.g., strict JSON Schema) will need an update. Lenient parsers ignore unknown fields. |
| `scan_id` changes from `"live"` to timestamp | ⚠️ | Breaks nothing functionally but changes the identifier format. Consumers that pattern-match on scan_id will need an update. |

## Future Possibilities

- **Hotspot trend CLI**: A `--hotspot-trend <path>` flag that reads
  `history.jsonl`, computes per-file hotspot score over time, and
  outputs a JSON array of (date, score) pairs for plotting.
- **IDE integration**: VSCode extension reads `hotspot.json` and
  decorates file tree nodes with zone emoji.
- **CI exit policy**: The existing code already exits 1 if any file
  is in the red zone (`has_red` check in main.zig). The hotspot.json
  makes this trivially queryable by CI runners without parsing stderr.
