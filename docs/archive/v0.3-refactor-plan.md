## 2. Proposed Changes

### 2.1 `src/report.zig` — Serialization helpers extraction

**Problem**: The file has grown organically. `writeJsonReport` contains a single `w.print()` call with 15+ format arguments (line 84) that is nearly unreadable. `writeMarkdownReport` is a single ~160-line function mixing summary statistics, zone tables, coupling tables, and expandable legend sections. JSON escaping helpers (`jsonEscape` / `jsonEscapeEscaped`) are duplicated across this file and `main.zig`.

**Plan**:

1. **Extract `writeFileResultJson`** — a helper that serializes a single `FileResult` as a JSON object. Used by both `writeJsonReport` and the inline hotspot.json writer in `main.zig`. This eliminates the 15-arg format string.

2. **Extract `writeCouplingPairJson`** — a helper that serializes a `CouplingPair`. Used by both `writeJsonReport` (couplings array) and `writeCouplingJson`.

3. **Factor `writeMarkdownReport` into sections:**
   - `writeMarkdownSummary(w, report)` — zone counts table
   - `writeMarkdownLegend(w)` — the `<details>` expandable section
   - `writeMarkdownFileTable(w, files)` — the file-by-file zone table
   - `writeMarkdownCriticalSection(w, files)` — red/orange listings
   - `writeMarkdownCouplingSection(w, couplings)` — coupling table + legend

4. **Extract shared JSON escaping** — move `jsonEscape` and `jsonEscapeEscaped` into their own private section (or a tiny `json.zig` helper). Ensures `main.zig` doesn't need to call `report.jsonEscapeEscaped`.

**Expected outcome**: `writeJsonReport` drops from 70 lines to ~30. `writeMarkdownReport` drops from 160 lines to ~30 (delegating to section helpers). Each helper is testable independently. The 15-arg format string disappears.

### 2.2 `src/coupling.zig` — Merge duplicate passes

**Problem**: `computeCoupling` makes **two near-identical passes** over `commit_files_list` (lines 54–100 and 113–141). The only difference is the target map: the first pass writes to `shared_map` + `total_commits_map`, the second writes to `first_half`/`second_half`. The file-indexing logic (`path_to_idx` / `file_idxs` building) is copy-pasted. The packed u64 key encoding appears 3 times.

**Plan**:

1. **Extract `indexCommitFiles`** — a helper that, given a commit, returns the list of file indices (using `path_to_idx` / `path_table`). Reused by both passes.

2. **Merge passes into one** — iterate once, but for each pair update three maps: `shared_map`, `first_half` (if `commit_idx < half`), `second_half` (otherwise). This eliminates the second pass entirely.

3. **Extract `packPairKey` / `unpackPairKey`** — small inline helpers for the `(min << 32) | max` encoding. Clearer than repeating the bit ops.

**Expected outcome**: `computeCoupling` drops from 200 lines to ~120. No duplicate iteration. The comment "We need two-pass" on line 93 becomes history.

### 2.3 `src/main.zig` — Extract inline blocks

**Problem**: `main` is a ~360-line function with 10 labeled pipeline steps and 5 inline blocks (`{ ... }`) that contain substantial logic:
- Hotspot.json building (lines 219–266, ~50 lines) — inlines JSON serialization + sort + write
- Stdout summary (lines 279–307, ~30 lines) — inlines counting + formatting
- History DB filtering (lines 353–421, ~70 lines) — inlines date parsing + line filtering + write
- File results merging (lines 164–211, ~50 lines) — inlines path lookup + trend compute + signal compute

**Plan**:

1. **Move hotspot.json writing into `report.zig`** — add `pub fn writeHotspotJson(w: anytype, report_data, thresholds, files) !void` to `report.zig`. It already contains `writeThresholdsJson`, `writeCouplingJson`, etc. — this is a natural fit.

2. **Extract `writeStdoutSummary`** — move the stdout printing block into a private helper in `main.zig`.

3. **Extract `mergeFileResults`** — move the path→evolution lookup + signal computation into `fn mergeFileResults(a, file_metrics_list, time_series_list, evolution_list) !std.ArrayList(types.FileResult)` in `main.zig`.

4. **Extract `filterAndWriteHistory`** — move the history DB filtering + writing into `fn filterAndWriteHistory(a, io, repo_dir, config, file_results, scan_id) !void` in `main.zig`.

5. **Simplify `main` to a clear pipeline** — each step becomes a single function call. The function shrinks from ~360 lines to ~80 lines (mostly pipeline calls + output blocks).

**Expected outcome**: `main` becomes readable as a high-level pipeline. Each extracted function is testable in isolation.

### 2.4 `src/git_log.zig` — Shared git-execution helper

**Problem**: `runGitLog` and `runGitLogNameOnly` both spawn git, capture stdout/stderr via a child process, and check the exit code. The boilerplate is almost identical (~30 lines each).

**Plan**:

1. **Extract `runGitAndCapture(a, io, argv, verbose) !struct{stdout: []u8, stderr: []u8}`** — shared helper that spawns, captures, checks exit code, and returns the output.

2. **Simplify both `runGitLog` and `runGitLogNameOnly`** to build argv then call the helper.

**Expected outcome**: ~30 lines of duplicated boilerplate eliminated. Both functions focus on parsing, not process management.

---

### 2.5 `src/scan.zig` — Extract directory-opening helper

**Problem**: The `if absolute / else relative` branching for opening the repo directory appears in both `scan.zig` (lines 18–22) and `main.zig` (lines 115–120).

**Plan**:

1. **Extract `openRepoDir(io, path) !std.Io.Dir`** into a shared private helper. Could live in `main.zig` or a new `dir.zig`, but simplest is to add it to `types.zig` as a utility.

**Expected outcome**: Duplicate branching pattern eliminated. One place to fix if the API changes.

---

### 2.6 `src/ignore.zig` — Additive tests only

**Problem**: The `matchesGlob` function is inherently complex (backtracking + star-position tracking). Complexity 7.2 (orange) is reasonable for what it does.

**Plan**: No structural changes. Add edge-case tests for `**/` prefix patterns and trailing `/` patterns common in `.gitignore`-style ignore files.

## 3. Implementation Order

```
Step A: src/report.zig   — Extract helpers (writeFileResultJson, section funcs)
Step B: src/main.zig     — Move hotspot.json into report.zig, extract helpers
Step C: src/coupling.zig — Merge duplicate passes, extract helpers
Step D: src/git_log.zig  — Extract shared git execution helper
Step E: src/scan.zig     — Extract openRepoDir utility
Step F: src/ignore.zig   — Add edge-case tests (no structural changes)
```

Within each step:

1. Extract helper function (preserves the original call site via delegation).
2. Run `zig build test` — all existing tests must pass.
3. Verify `zig build` — the binary compiles.
4. Run a full scan (e.g. `zig build run -- .`) — output must match the current behavior.

## 5. Risk Assessment

| Change | Risk | Mitigation |
|--------|------|------------|
| Extract helpers in `report.zig` | Low — pure code motion, no logic change | Tests verify roundtrip |
| Merge coupling passes | Medium — algorithmic change | Before/after diff on real repos (multiple test cases) |
| Move hotspot.json to `report.zig` | Low — same serialization, different location | Tests verify output structure |
| Extract pipeline helpers from `main.zig` | Low — code motion | Tests verify each helper |
| Change git execution | Low — the argv + parsing stays the same | Tests cover both git functions |

### Rollback plan

Each step is a self-contained commit. If a step introduces a regression, revert that single commit. Steps A–F are designed to be independent (no circular dependencies), so partial rollback is safe.

## 6. Success Criteria

After all steps are complete:

1. **`zig build test` passes** — all existing + new tests.
2. **`zig build run -- .` produces identical JSON output** (byte-for-byte, keys sorted) compared to the pre-refactor version.
3. **No red zone files remain.** Specifically:
   - `src/report.zig` hotspot drops below 🔴 threshold (14.2)
   - `src/coupling.zig` complexity drops below 🟠 threshold (5.4)
   - `src/main.zig` complexity drops below 🔴 threshold (7.7)
4. **Code review**: Each extracted function has a clear name, doc comment, and single responsibility.
